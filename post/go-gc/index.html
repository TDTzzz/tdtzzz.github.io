概述

垃圾回收（英语：Garbage Collection，缩写为GC）是指一种自动的内存管理机制。当某个程序占用的一部分内存空间不再被这个程序访问时，这个程序会借助垃圾回收算法向操作系统归还这部分内存空间。垃圾回收器可以减轻程序员的负担，也减少程序中的错误

常见的垃圾回收方法

1.引用计数

引用计数通过在对象上增加自己被引用的次数，被其他对象引用时加1，引用自己的对象被回收时减1，引用数为0的对象即为可以被回收的对象。这种算法在内存比较紧张和实时性比较高的系统中使用的比较广泛，如ios cocoa框架，php，python等。

优点：

  1. 方式简单，回收速度快。

缺点：

  1. 需要额外的空间存放计数。

  2. 无法处理循环引用（如a.b=b;b.a=a这种情况）。

  3. 频繁更新引用计数降低了性能

2.标记清除法(mark and sweep)

goV1.3之前所用的gc方法

流程

  1. 暂停程序业务逻辑，找出不可达的对象和可达对象
  2. 开始标记，程序找出它所有可达的对象，并做上标记
  3. 标记完了之后，开始清除未标记的对象
  4. 停止STW，让程序继续跑，然后循环这个过程，直到process程序生命周期结束

缺点

  1. STW,stop the world;让程序暂停，程序出现卡顿
  2. 标记需要扫描整个heap
  3. 清除数据会产生heap碎片

3.三色标记法

go在1.5版本改为了三色标记法，这种方法的mark操作可以渐进执行而不是需要每次都扫描整个内存空间，可以减少STW的时间。

go在1.8版本又改动了gc，采用了混合写屏障（hybrid write barrier）来避免堆栈重新扫描。

对象丢失的现象（不STW保护的话）

两个条件同时满足则会出现对象丢失

  1. 一个白色对象被一个黑色对象引用（白色被挂在黑色下）
  2. 灰色对象与他们之间的可达关系的白色对象遭到破坏（灰色同时丢了白色）

例子如下

  1. 已经被标记成灰色的对象2有指针p指向白色对象3
  2. 在还没扫描到对象2，已经标记为黑色的对象4闯将指针q指向对象3
  3. 与此同时灰色对象2将指针p移除，白色对象就只被挂在已经扫描完成的黑色对象4下了
  4. 正常执行算法逻辑，灰色对象2,7被标记为黑色，而白色对象3因为黑色对象已经不会再被扫描，所以被等待清除。

屏障机制

插入屏障

对象被引用时触发的机制

满足

满足强三色不变式（不存在白色被黑色引用了）

场景
  1. A之前无下游，新添加一个夏有对象B，B被标记成灰色
  2. A将下游对象由C更换成B，B被标记成灰色
不足

结束时需要STW来重新扫描栈，大约需要10~100ms

删除屏障

对象被删除时触发的机制

满足

满足若三色不变式（保护灰色对象到白色对象的路径不会断）

场景
  1. A对象，删除B对象的引用。B被A删除后被标记成灰（如果之前为白）
  2. A对象，更换下游B为C。B被A删除后被标记成灰（如果B之前为白）
不足

回收精度低，一个对象即是被删除了，最后一个指向它的指针也依然可以活过这一轮，在下一轮GC才能被清理掉。

混合写屏障机制

通俗来讲就是分堆空间和栈空间，栈上没屏障操作，添加还是删除都标价为黑色，实际上可以看做是一种延迟回收机制来避免STW

4.go的gc总结

  1. 1.3版本的普通标记清除法，整体过程需要STW,效率极低
  2. 1.5版本的三色标记法，对空间启动写屏障，栈空间不启动，全部扫描之后，需要重新扫描一次栈(需要STW)，效率一般
  3. 1.8版本的三色标记法，混合写屏障机制，栈空间不启动，堆空间启动柜，整体过程不需要STW(gc标记准备阶段和GC标记结束阶段还是需要STW 来通知所有的Goroutine 我进入垃圾回收阶段了,然后启用/关闭屏障机制,但是这个耗时很小,一般几十微妙)，效率较高
